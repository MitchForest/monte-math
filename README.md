# Monte Math Monorepo

Monte Math is a Montessori-inspired mathematics platform that combines programmatic materials, scripted lessons, and a contract-first API. This monorepo contains every part of the stack—from authored content to oRPC-backed services to the React applications that present lessons and authoring tools.

## High-Level Architecture

```
┌────────────┐     ┌────────────────────┐     ┌──────────────────────┐
│ content/   │ --> │ @monte/shared      │ --> │ apps/api (BFF/API)   │
│ (skills &  │     │  • Zod v4 schemas  │     │  • oRPC server       │
│ lessons)   │     │  • oRPC contract   │     │  • SQLite/Kysely      │
└────────────┘     └────────────────────┘     └──────────┬───────────┘
                                                        │
                                     ┌──────────────────┴───────────────────┐
                                     │           @monte/api-client          │
                                     │  • Wraps contract with RPC fetch     │
                                     │  • Shared by all client apps         │
                                     └───────────┬───────────┬──────────────┘
                                                 │           │
                                     ┌───────────▼───┐   ┌───▼──────────┐
                                     │ apps/web      │   │ apps/studio  │
                                     │ (Player)      │   │ (Authoring)  │
                                     └───────────────┘   └──────────────┘
```

### Key Conventions

- **Contract-first API:** Every request/response is described in `@monte/shared` using Zod 4. The Bun API implements the contract with `@orpc/server`, ensuring runtime validation and TypeScript safety end-to-end.
- **Single source of content:** Authored skills and lessons live in `content/`. The API seeds/reads from this folder, so local development and publishing share identical assets.
- **Strict package boundaries:**
  - `@monte/shared` contains only schemas/types—no fetch, no runtime side-effects.
  - `@monte/api-client` owns client creation (oRPC link + headers).
  - Applications depend on the client, never directly on API implementation files.
- **Styling system:** Tailwind CSS v4 + shadcn/ui (New York theme) everywhere. Each app imports `src/styles/globals.css` generated by the shadcn CLI.
- **Data flow:** Content → shared schemas → API (Bun/Kysely) → clients (React) via the oRPC contract. No component reaches into `.docs` or other app directories.

## Repository Layout

```
apps/
  api/       Bun + Hono-powered API implementing the oRPC contract
  web/       React player showcasing materials and lessons
  studio/    React authoring/graph explorer for curriculum teams

packages/
  shared/        Zod schemas, TypeScript types, and the oRPC contract
  content-builder/ (roadmap) CLI helpers for validating & publishing content
  api-client/    Typed client factory built on the shared contract

content/
  skills/graph.json                               Seed data for skills & prereqs
  lessons/<lesson-id>/script.json                 Immutable lesson scripts
```

## Data & Control Flow

1. **Author content** as JSON in the `content/` directory. Scripts are validated against `lessonScriptSchema` when loaded.
2. **Seed + query** through the API. The Bun server uses Kysely to read/write SQLite (`apps/api/src/db`) and validates every payload through Zod.
3. **Consume** via `@monte/api-client`. Both `apps/web` and `apps/studio` create a client instance (with custom headers) and interact only with contract-defined procedures.
4. **Render** lesson flows and knowledge graphs. React Query manages caching; stateful components (Pixi scenes, Zustand stores) stay inside their respective apps.

## Technology Choices

- **API:** Bun, Hono, Kysely (SQLite/Turso-ready). oRPC provides a light-weight RPC surface with OpenAPI compatibility while keeping strict typing.
- **Client Apps:** Vite + React 18, TanStack Router/Query, Zustand for local state, PixiJS for materials, Tailwind v4 + shadcn for UI primitives.
- **Validation:** Zod 4 from content to contract to API; no `any`/`unknown` escapes.
- **Styling/Design:** Tailwind token layer maintained in `globals.css`, with shadcn-generated components living per-app.

## Development Workflow

1. Install dependencies: `pnpm install`
2. Run database migrations: `pnpm --filter @monte/api db:migrate`
3. Seed skills (optional local): `pnpm --filter @monte/api db:seed`
4. Start the stack:

   ```bash
   pnpm dev:api     # http://localhost:3001 (RPC under /rpc)
   pnpm dev:web     # http://localhost:5173
   pnpm dev:studio  # http://localhost:3002
   ```

5. Sync canonical content and database:
   ```bash
   pnpm sync:pull   # Import content/ JSON into SQLite (via db:seed)
   pnpm sync:push   # Export published DB state back to content/
   ```

## Contributing Guidelines

- Keep cross-package dependencies minimal. If two apps need the same runtime code, extract it into a package.
- Only the API should touch the database or read directly from `content/`.
- Validate new content against the shared schemas—avoid ad-hoc JSON shapes.
- When adding an API endpoint, update the contract in `packages/shared`, regenerate the implementation in `apps/api`, and consume through `@monte/api-client`.
